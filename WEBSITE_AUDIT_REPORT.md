## 1. 项目概览

### 1.1 项目名称

向光而行 · 日期倒计时

### 1.2 项目定位

该网站是面向高中生的学习与考试日期倒计时网站。核心目标是：

- 在一个页面中清晰展示最多三个重要日期倒计时。
- 提供积极、温暖、有力量的定时句子。
- 保持克制、清爽、具有学习氛围和希望感的视觉风格。
- 不依赖复杂框架，可直接部署为静态网站。

- 网站：https://aaaaarry.github.io/countdown-to-gaokao/
- GitHub 仓库：https://github.com/Aaaaarry/countdown-to-gaokao
- 部署方式：GitHub Pages
- 部署分支：`main`
- 部署目录：仓库根目录 `/`
- HTTPS：已启用
- 当前部署状态：`built`

## 4. 倒计时功能

### 4.1 默认倒计时

默认配置为：

| 名称 | 日期 |
| --- | --- |
| 高考 | 2027年6月7日 |
| 首考 | 2027年1月6日 |
| 市统测 | 2026年9月1日 |

### 4.2 显示规则

- 最多同时显示三个倒计时。
- 三个倒计时统一显示在同一卡片内，不需要切换页面。
- 每项显示倒计时名称、剩余天数、单位“天”和目标日期。
- 已到达或超过目标日期时，天数不会显示为负数，而是显示 `0`。
- 原“这一程的进度”进度条已删除。

### 4.3 日期计算

- 日期差按日历日期计算，而不是简单用小时数除以24。
- 使用本地年月日并转换为 UTC 日历值，避免夏令时切换造成一天误差。
- 页面每分钟重新计算一次，跨过午夜后会自动更新天数。

### 4.4 自定义与排序

点击“管理倒计时”可以：

- 添加自定义倒计时名称。
- 选择目标日期。
- 编辑已有倒计时名称和日期。
- 删除倒计时。
- 使用上移、下移按钮调整顺序。

限制：

- 最多三个倒计时。
- 至少保留一个倒计时。

### 4.5 本地保存

自定义倒计时使用 `localStorage` 保存。

这意味着：

- 同一浏览器再次打开网站时会保留设置。
- 更换浏览器、设备或清除浏览器数据后不会保留。
- 不同访问者拥有各自独立的设置。

## 5. 句子定时刷新功能

### 5.1 刷新方式

- 页面打开时立即显示一条句子。
- 每5分钟自动刷新一条句子。
- 点击“换一句”可以立即刷新。
- 手动刷新也遵循未展示优先规则。

### 5.2 月度句子库结构

句子数据采用以下结构：

```js
月份（YYYY-MM）
  └── 星期（0=星期日，1=星期一，...，6=星期六）
        └── 句子数组
```

示例：

```js
const MONTHLY_QUOTE_LIBRARIES = {
  "2026-06": {
    0: [],
    1: [],
    2: [],
    3: [],
    4: [],
    5: [],
    6: [],
  },
};
```

网站根据用户设备的当前月份和星期自动选择对应句子数组。

### 5.3 当前数据量

当前是逻辑演示版本，不是完整700条/月版本。

- 已配置月份：`2026-06`、`2026-07`。
- 每个星期当前提供5条月度示例。
- 另有独立应急句子库和旧版补充句子。
- 当前代码中共定义86条句子。
- 经检查，86条句子文本全部唯一。

正式数据目标：

- 每个星期100条。
- 每个月7个星期分类，共700条。
- 新月份与上月份内容不重复。

### 5.4 当天未展示优先

网站通过 `localStorage` 保存：

```text
当天日期 + 已展示句子索引
```

规则：

1. 每次刷新优先选择当天尚未展示的句子。
2. 所有句子展示完毕后，才开始下一轮。
3. 新一轮避免立即重复当前句子。
4. 日期变化后自动重置当天展示历史。

### 5.5 连续4小时不重复保证

刷新间隔为5分钟。

如果包含页面打开时立即显示的第一条，则从第0分钟到第240分钟共有49次展示。因此每日保护池至少包含49条互不重复句子。

当前保护池构建规则：

1. 优先使用当天星期对应的句子。
2. 当前示例不足49条时，从当月其他星期的句子中补充。
3. 继续使用该月份专属的补充分片补足。
4. 去除所有文本重复项。
5. 最终截取至少49条。

因此可以保证：

- 页面初次展示至完整4小时结束，定时刷新不会重复。
- 点击“换一句”也优先使用保护池内未展示内容。

注意：频繁点击“换一句”会更快消耗保护池。若在4小时内手动切换超过保护池容量，全部句子用完后会开始下一轮。

当前月度基础示例库之间已经无重复。已配置月份使用互不重叠的补充分片，程序会对运行时保护池执行跨月检查。长期扩展到更多月份时，仍建议正式补齐每月700条独立句子并停止使用补充内容。

### 5.6 英文句子显示

英文句子支持以下字段：

- 英文原文。
- 较大字号的中文翻译。
- 英文人物姓名。
- 中文人物姓名。
- 几个字的身份简介。

示例：

```js
{
  text: "It always seems impossible until it's done.",
  translation: "在事情完成之前，它看起来总是不可能。",
  source: "Nelson Mandela · 纳尔逊·曼德拉",
  bio: "南非前总统、反种族隔离领袖"
}
```

没有明确作者的原创句子显示为“今日寄语”，不会再将“月份·星期”分类标签误显示成作者。

### 5.7 每月切换与缺月处理

- 每月1日，网站根据新日期自动选择新月份句子库。
- 页面每分钟检查一次日期，跨过午夜后立即切换。
- 如果当前月份尚未配置，不会回退并重复上个月句子。
- 缺月时使用独立应急句子库，并在浏览器控制台提示维护人员添加新月份。

### 5.8 自动校验

网站启动时会在浏览器控制台执行以下检查：

- 月内是否存在重复句子。
- 相邻配置月份之间是否存在重复句子。
- 英文句子是否缺少中文翻译、人物中文名或简介。
- 每个星期是否达到正式目标100条。
- 当前每日保护池是否达到49条。
- 应急句子库是否与月度库重复。

## 6. 纯前端与后端/API版本

### 6.1 当前纯前端版本

当前网站是纯前端静态版本。

优点：

- 部署简单。
- 无服务器费用。
- GitHub Pages即可运行。
- 页面加载后不依赖外部接口。

限制：

- 新月份句子需要人工写入 `quotes-data.js`。
- 修改句子后需要重新发布网站。
- 无法在不同设备之间同步倒计时设置和句子展示历史。
- 无法由服务器统一审核和实时修正句子。

### 6.2 推荐的后端/API版本

建议接口：

```text
GET /api/quotes/daily?date=2026-06-11
```

建议返回：

```json
{
  "date": "2026-06-11",
  "month": "2026-06",
  "weekday": 4,
  "version": "2026-06-v1",
  "quotes": []
}
```

建议后台流程：

1. 每月25日生成或导入下个月700条句子。
2. 检查月内重复、跨月重复、翻译和来源。
3. 人工审核。
4. 审核通过后发布新版本。
5. 前端每月1日自动获取新月份内容。
6. API不可用时使用浏览器缓存或应急库。

后端版本需要数据库、定时任务、接口鉴权和运维。

## 7. 响应式与体验

- 桌面端为左右双栏布局。
- 手机端为上下单列布局。
- 使用原生按钮和表单。
- 重要按钮具有可点击区域和悬停反馈。
- 句子切换使用轻微淡入淡出。
- 支持系统“减少动态效果”设置。
- 使用 `aria-live` 让倒计时和句子更新更容易被辅助技术感知。

## 8. 数据与隐私

- 网站不要求登录。
- 不收集姓名、邮箱或学习记录。
- 不向服务器上传用户自定义倒计时。
- `localStorage` 中只保存：
  - 用户自定义倒计时。
  - 当天已展示句子的索引。
- GitHub Pages会按照GitHub自身政策记录常规访问日志。

## 9. 当前已知限制与风险

### 9.1 句子数据尚未达到正式规模

当前每个星期只有5条月度示例，尚未达到每天100条、每月700条的内容目标。四小时不重复通过保护池和补充分片实现。

审核建议：正式投入长期使用前，补齐每月700条经过审核的内容。

### 9.2 内容版权与准确性

文学摘录、名人名言和人物归属需要进一步核实版权、原文和出处准确性。

审核建议：

- 优先使用公版内容、自有原创内容或获得授权的内容。
- 对英文翻译和人物简介进行人工校对。
- 不确定出处的句子标记为“今日寄语”，不要错误归属名人。

### 9.3 运行时补充池容量有限

当前配置的六月和七月基础月度示例互不重复，使用的补充分片也互不重叠。但旧版补充句子总量有限，继续增加大量月份后将不足以为每个月提供独立补充分片。

审核建议：正式补齐每月700条独立内容后，移除运行时补充机制。

### 9.4 本地存储不同步

用户数据仅存在当前浏览器。

审核建议：如需要跨设备同步，增加账号与后端数据库。

### 9.5 月度数据需要人工维护

纯前端版本无法真正自动生成下个月数据。

审核建议：建立月度内容更新责任人和发布时间表，或开发后端/API版本。

## 11. 审核结论摘要

当前网站的视觉布局、倒计时管理、响应式显示、月度星期句子结构、每日未展示优先和连续4小时不重复保护机制均已实现并部署。

- 需要从当前每天5条示例扩充到每天100条。

- 需要建立每月更新与发布流程，或升级为后端/API自动更新版本。
