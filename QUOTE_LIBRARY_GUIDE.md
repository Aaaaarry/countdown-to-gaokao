# 月度句子库维护与 API 设计

## 当前纯前端版本

句子数据位于 `quotes-data.js`，结构为：

```js
const MONTHLY_QUOTE_LIBRARIES = {
  "2026-06": {
    0: [], // 星期日，正式使用时放入 100 条
    1: [], // 星期一
    2: [],
    3: [],
    4: [],
    5: [],
    6: [], // 星期六
  },
};
```

- 每个月应包含星期 `0` 至 `6` 的七个数组。
- 每个数组正式使用时放入 100 条互不重复的句子，共 700 条。
- 每月月底前，在 `quotes-data.js` 中手动增加下个月数据。
- 网站会根据用户设备的本地月份和星期自动选择对应数组。
- 当月数据不存在时，会使用独立应急句子库，不会回退并重复上个月内容，同时在控制台提示维护人员添加新月份。
- 网站启动时会检查月内重复、与上一个配置月份的重复，以及每天是否达到 100 条。
- `localStorage` 只保存当天日期和已展示的索引；日期变化后自动重置。
- 当前示例每天只有 5 条，因此程序会从当月其他星期及该月专属补充分片中构建至少 49 条的保护池。包含页面初次展示后，0 至 240 分钟共有 49 次展示，可保证完整连续 4 小时不重复。
- 正式将每天数组扩充到 100 条后，保护池只使用当天的 100 条句子，不再需要补充内容。

纯前端优点是部署简单、无需服务器、没有接口费用。缺点是无法真正自动生成或更新新月份数据，每次更新句子库后需要重新发布网站。

## 后端 / API 版本建议

后端版本可将句子保存在数据库，由定时任务在每月结束前生成、审核并发布下月句子库。

建议接口：

```text
GET /api/quotes/daily?date=2026-06-11
```

响应：

```json
{
  "date": "2026-06-11",
  "month": "2026-06",
  "weekday": 4,
  "version": "2026-06-v1",
  "quotes": [
    {
      "id": "2026-06-4-001",
      "text": "句子内容",
      "translation": "",
      "source": "来源",
      "bio": ""
    }
  ]
}
```

后台定时任务建议流程：

1. 每月 25 日生成或导入下个月七天分类的句子，每天至少 100 条。
2. 对本月、上月和历史句子做文本规范化与重复检查。
3. 对引用来源、英文翻译和人物介绍进行人工审核。
4. 审核通过后发布新版本，前端在每月 1 日自动请求新月份数据。
5. API 不可用时，前端继续使用上一次缓存的数据。

后端版本优点是能够自动更新、集中审核和随时修正内容，无需重新发布前端。缺点是需要服务器、数据库、定时任务、鉴权和运维成本。
